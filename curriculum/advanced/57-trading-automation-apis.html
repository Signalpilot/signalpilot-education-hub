  • Order management is complex‚Äîduplicates, rate limits, error handling: Mike's March bug: API returned "pending," bot assumed "no fill," resent order 47 times. Mike's 7-Step Production-Ready Automation Framework

Step 1: Paper Trading Phase (2-4 Weeks Minimum)

Rule: NEVER deploy bot to live trading without 2-4 weeks of paper trading first. Run bot on sim account, monitor for: (1) Unexpected behavior (duplicate orders, excessive API calls, wrong position sizes), (2) Error handling (what happens when API returns error? When order rejected?), (3) Slippage reality (paper fills show real slippage vs backtest assumptions). Only after 50-100 paper trades with ZERO critical bugs ‚Üí move to live with tiny size.

Step 2: Multi-Layer Kill Switches

Rule: Implement 5 circuit breakers BEFORE live trading: (1) Max daily loss (-2% = stop all trading for day), (2) Max drawdown (-8% from peak = pause, review strategy), (3) Volatility filter (VIX >30 or ATR spike >150% = reduce size 50% or stop), (4) Max open positions (never >5 positions simultaneously, prevents overconcentration), (5) Remote kill switch (SMS command "STOP" immediately halts bot, closes all positions). Test all 5 kill switches on paper account before live.

Step 3: Order Management System

Rule: Track every order with unique ID. Before sending order: Check if order for same symbol+side already pending (prevent duplicates). After sending: Store order ID, poll for status every 2-5 seconds, only mark "filled" after API confirms. Implement retries with exponential backoff (if API error, wait 2s, retry; if error again, wait 4s, retry, etc.). Never assume order filled without confirmation. Never send duplicate order because "API slow to respond."

Step 4: Rate Limit Management

Rule: Stay at 70-80% of API rate limit max. If limit is 200 req/min, self-limit to 150 req/min. Implement request queue (if bot wants to make 10 requests simultaneously, queue them, send at controlled rate). Add exponential backoff on 429 errors (rate limit hit = wait 30s before retrying, doubles each time). Monitor daily API usage (most brokers have daily limits too). Getting API-banned = missing trades = opportunity cost. Prevention is cheaper than cure.

Step 5: Walk-Forward Optimization (Not Curve-Fitting)

Rule: Never optimize params on full dataset then deploy. Use walk-forward method: (1) Train on Year 1 data, test on Year 2, (2) Train on Year 2 data, test on Year 3, (3) Repeat. Only deploy if strategy profitable across ALL walk-forward periods (proves it works in different regimes, not just optimized for one year). Mike's mistake: Optimized for 2023, failed in 2024. Walk-forward would've shown: "These params work in 2023 but fail in 2022 and early-2024 ‚Üí don't deploy."

Step 6: Slippage/Fees in Backtest

Rule: Add realistic slippage to EVERY backtest trade. Conservative estimate: -0.05% per trade for limit orders, -0.15% for market orders. Add fees: $0.50-2.50/trade depending on broker. Subtract these from every trade in backtest. If strategy still profitable after slippage+fees ‚Üí it's real. If it becomes unprofitable ‚Üí it was illusion. Mike's error: Backtest showed 74% WR (zero slippage), live showed 48% WR (real slippage). Adding -0.15% slippage to backtest would've predicted 58-62% live WR (closer to reality).

Step 7: Monitoring Dashboard + Alerts

Rule: Bot runs 24/7, you can't watch 24/7. Set up: (1) SMS alerts for critical events (daily loss >-1.5%, position loss >-3%, API errors >5/hour, bot offline >5min), (2) Dashboard showing: open positions, daily P&L, win rate today, API health, last trade time, (3) Daily summary email: trades taken, P&L, any errors, system health. Check dashboard 2-3√ó/day minimum. Never "set and forget"‚Äîautomation requires MONITORING, not absence.

Step 3: Detect Sweep Potential Reversal Setup

def get_swing_low(bars, lookback=20):
    """Find swing low over last N bars"""
    lows = [bar.l for bar in bars[-lookback:]]
    return min(lows)

def check_sweep(current_price, current_low, swing_low):
    """
    Sweep conditions:
    1. Current low swept below swing low (by 0.3%)
    2. Price reclaimed back above swing low
    """
    if swing_low is None:
        return False

    # Check if swept below
    if current_low < swing_low * 0.997:  # 0.3% below
        # Check if reclaimed
        if current_price > swing_low:
            return True  # Sweep confirmed!

    return False

This is your edge, coded. If conditions = true, trade. If false, skip.

No hesitation. No emotion. Just logic.

Step 4: Calculate Position Size (Risk Management)

def calculate_position_size(api, potential entry, stop):
    """
    Risk 2% of account per trade
    Position size = (Account √ó Risk%) / (Entry - Stop)
    """
    account = api.get_account()
    equity = float(account.equity)
    risk_amount = equity * 0.02  # 2% risk

    stop_distance = abs(potential entry - stop)
    size = int(risk_amount / stop_distance)

    return size

# Example: $100K account, $520 potential entry, $518 stop
# Risk: $100K √ó 2% = $2,000
# Stop distance: $520 - $518 = $2
# Size: $2,000 / $2 = 1,000 shares

Your bot now risks exactly 2% per trade. Every time. No exceptions.

Step 5: Place Bracket Order (Entry + Stop + Target)

def place_bracket_order(api, symbol, size, potential entry, stop, target):
    """
    Bracket order = potential entry + stop-loss + take-profit
    All three orders placed simultaneously
    """
    try:
        order = api.submit_order(
            symbol=symbol,
            qty=size,
            side='buy',
            type='limit',
            limit_price=potential entry,
            time_in_force='gtc',  # Good till canceled
            order_class='bracket',
            stop_loss={'stop_price': stop},
            take_profit={'limit_price': target}
        )

        print(f"‚úì Order placed: {size} shares at ${potential entry}")
        print(f"  Example stop: ${stop} | Example target: ${target}")
        return order

    except Exception as e:
        print(f"‚úó Order failed: {e}")
        return None

One function call. Three orders placed. Your risk is defined before you're even filled.

Step 6: Put It All Together (The Main Loop)

import time

def run_bot():
    """Main bot loop"""
    swing_low = None
    position = None

    print("Bot started. Watching for sweep setups...")

    while True:
        try:
            # Fetch latest data
            barset = api.get_barset('SPY', 'minute', limit=50)
            bars = barset['SPY']
            current_bar = bars[-1]
            current_price = current_bar.c
            current_low = current_bar.l

            # Update swing low
            swing_low = get_swing_low(bars)

            # Check if we're already in a position
            if position is None:
                # Look for setup
                if check_sweep(current_price, current_low, swing_low):
                    print(f"\nüéØ Sweep detected at ${current_price}!")

                    # Calculate potential entry, stop, target
                    potential entry = current_price
                    stop = swing_low * 0.995  # Below sweep
                    target = potential entry + (potential entry - stop) * 3  # 3R target

                    # Calculate size
                    size = calculate_position_size(api, potential entry, stop)

                    # Place trade
                    position = place_bracket_order(
                        api, 'SPY', size, potential entry, stop, target
                    )

            # Sleep 60 seconds (check every minute)
            time.sleep(60)

        except Exception as e:
            print(f"Error: {e}")
            time.sleep(60)  # Continue despite errors

# Run the bot

Common Errors (And How to Handle Them)

Things That Will Go Wrong

  • Connection timeout: Internet hiccup, API down
  • Order rejected: Insufficient funds, invalid price
  • Partial fills: Order only 60% filled
  • Rate limit exceeded: Too many API calls
  • Market closed: Tried to trade at 3 AM

"It worked yesterday! Why is it broken today?!"

Bulletproof Your Bot

  • Retry logic: Try 3 times with exponential backoff
  • Pre-trade checks: Verify funds, market hours, valid price
  • Partial fill handling: Cancel remaining or market-fill
  • Rate limiting: Max 200 requests/minute (add delays)
  • Market hours check: Only trade 9:30 AM - 4:00 PM ET

Professional bots handle errors gracefully, log everything, and keep trading.

Retry Logic (Essential)

def place_order_with_retry(api, **order_params):
    """Retry order placement up to 3 times"""
    max_retries = 3

    for attempt in range(max_retries):
        try:
            order = api.submit_order(**order_params)
            return order  # Success!

        except Exception as e:
            if attempt < max_retries - 1:
                wait = 2 ** attempt  # 1s, 2s, 4s (exponential backoff)
                print(f"Order failed: {e}. Retry in {wait}s...")
                time.sleep(wait)
            else:
                # Final attempt failed
                print(f"Order failed after {max_retries} attempts: {e}")
                # Send alert (email, SMS)
                raise  # Re-raise exception

import smtplib
from email.mime.text import MIMEText

def send_alert(subject, message):
    """Send email alert"""

    msg['Subject'] = subject
    msg['From'] = 'bot@yourbot.com'
    msg['To'] = 'you@gmail.com'

    # Send via Gmail SMTP
    with smtplib.SMTP('smtp.gmail.com', 587) as server:
        server.starttls()
        server.login('bot@yourbot.com', os.getenv('EMAIL_PASSWORD'))
        server.send_message(msg)
