Your Indicators Are Lying to You (The Repaint Problem)

Reading time ~15 min • Indicator Truth

60-90% of indicators repaint—showing perfect signals in backtest that never existed real-time. Learn to detect repainting, audit any indicator, and why Signal Pilot's deterministic protocol matters.

What IS Repainting?

Definition: An indicator that changes its past values or signals after the bar closes. What you saw live disappears or moves in historical view.

Example: Indicator shows "BUY" signal at $100. You enter. Bar closes. You check history—signal is now at $95. You were tricked.

Why Repainting Happens

1. Future Data Leakage

request.security() without lookahead=barmerge.lookahead_off pulls future data into current bar.

2. Non-Deterministic Calculations

Indicators that recalculate based on new data (changing past values).

3. HTF (Higher Timeframe) Repainting

HTF bar hasn't closed yet, but indicator uses its incomplete data.

The 4 Methods to Detect Repainting

Method 1: Visual Test

  1. Note indicator signal on live chart
  2. Wait for bar to close
  3. Refresh chart
  4. Check if signal moved or disappeared

If it changed = REPAINTING

Method 2: Replay Test

  1. Use TradingView Bar Replay
  2. Mark where signal appears
  3. Finish replay
  4. Compare historical vs. live signals

Method 3: Alert Test

  1. Set alert on indicator signal
  2. Track alerts over 20+ trades
  3. Compare alert price vs. historical signal price

If they don't match = REPAINTING

Method 4: Code Audit

Red flags in Pine Script:

  • request.security() without lookahead_off
  • var misuse (changing historical values)
  • security() pulling HTF data without offset
  • No [1] offset for HTF confirmation

Safe code example:

// NON-REPAINTING:
htf_close = request.security(syminfo.tickerid, "D", close[1],
             lookahead = barmerge.lookahead_off)

Repainting code example:

// REPAINTING (bad):
htf_close = request.security(syminfo.tickerid, "D", close)

Signal Pilot's Deterministic Protocol

All Signal Pilot indicators follow strict rules:

  • No future data: All HTF requests use lookahead_off
  • Fixed signals: Once a signal prints, it never moves
  • Confirmed bars only: Signals wait for bar close (no intra-bar changes)
  • Open-source auditable: Code available for review

Complete Indicator Audit Framework

For any indicator you use:

  1. Visual test: Watch live, compare to history (3+ signals)
  2. Replay test: Use Bar Replay to verify signals
  3. Alert test: Set alerts, track 20+ instances
  4. Code audit: Check for request.security() and lookahead settings

If ANY test fails → REPAINTING → DO NOT USE

Why This Matters

Backtest: 80% win rate, 3R average winner (looks amazing!)

Reality: Signals repainted, actual results: 45% win rate, -0.5R expectancy (disaster)

You just wasted months developing a strategy that never worked.

Key Takeaways

  • 60-90% of indicators repaint (test everything)
  • Repainting = fake edge (backtest perfection, live failure)
  • 4 detection methods: Visual, Replay, Alert, Code Audit
  • Signal Pilot = deterministic (what you see is what you get)
  • Never trust, always verify (audit every indicator)
